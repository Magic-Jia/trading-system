# Trading System 多粒度收益优化设计（2026-04-06）

## 1. 背景与目标

当前 `trading_system` 已具备 `regime`、`trend`、`rotation`、`short`、`allocator`、`risk gate`、`paper execution` 与 `reporting` 的主链路，但最近的策略层回测已经暴露出几个清晰事实：

- `trend`（趋势）腿是当前唯一明确正贡献主线。
- `short`（做空）腿在现有实现下系统性拖累收益。
- `rotation`（轮动）并非没有价值，但被当前 `suppression`（抑制）与统一分配逻辑压得过重。
- 当前系统可以研究，但样本外稳定性不足，不能把现阶段结果视为稳定实盘 edge（优势）。

本设计的目标不是单纯做“更稳但赚更少”的保守版，而是在**不接受总收益明显下降**的前提下，优先提升：

1. 风险调整后收益
2. 最大回撤控制
3. 资金曲线平滑度
4. 样本外稳定性

## 2. 用户已确认的约束

本次设计基于以下已确认约束：

- 主目标：提高风险调整后收益，而不是只冲名义总收益。
- 不能接受“更稳但总收益明显变差”的方案。
- `short` 不彻底删除，但只在明确差环境里保留。
- 可以接受交易次数明显减少，换更高质量的开仓。
- 接受升级到底层数据粒度，不强行限制在现有日频框架内。

## 3. 设计结论（一句话版本）

把系统升级为：

- **日频（daily）负责方向、环境、候选与预算**
- **小时级（1h / 4h）负责入场确认、过热过滤、拥挤过滤、退出与利润保护**

也就是一个“**日频决策层 + 小时级执行层**”的双层结构。

## 4. 为什么选择多粒度混合版

相较于“只在日频内小修小补”或“直接全量重构整套系统”，多粒度混合版更适合当前阶段，原因如下：

1. **保留现有日频框架的稳定性与解释性**：当前系统的大框架、组合控制与 `regime` 逻辑已经成型，不需要推倒重来。
2. **把最伤收益 / 回撤比的问题交给更合适的粒度解决**：很多问题不是日频方向错，而是开仓太晚、追在局部扩张后面、退出太慢。
3. **更符合当前已知回测事实**：`trend` 值得继续做主赚钱腿；`rotation` 值得从“压死”改为“有条件放行”；`short` 需要退回到条件性工具。
4. **工程成本和研究价值比更高**：比大重构更快落地，比只调参数更有机会同时改善收益和回撤。

## 5. 总体架构

### 5.1 分层结构

系统升级后拆成 6 层：

1. **日频环境层**
   - 识别市场状态、风险开关、各策略腿预算
   - 决定 `rotation` 与 `short` 是否允许参与、以及参与上限

2. **日频候选层**
   - 继续生成 `trend` / `rotation` / `short` 候选
   - 保持现有大框架，不在本层处理精细 timing（时机）

3. **小时级确认层**
   - 对日频候选做二次确认
   - 重点处理：追涨过热、拥挤交易、短线位置、失败风险

4. **分腿分配层**
   - 改为 `trend` / `rotation` / `short` 三条独立预算通道
   - 由环境层动态约束各腿上限

5. **小时级退出层**
   - 对所有已开仓头寸提供更快、更细的退出与保护逻辑
   - 不再主要依赖日频慢反应处理失败与止盈

6. **统一回测评估层**
   - 强制做多版本对照、分腿归因与滚动验证
   - 防止“看起来更合理”但其实没有统计价值

### 5.2 责任边界

#### 日频层负责什么

- 当前是否适合积极做多
- 哪些币进入候选池
- 哪条策略腿拿多少预算
- `short` 是否允许启用
- `rotation` 是关闭、限额放行还是正常放行

#### 小时级层负责什么

- 是否值得现在开仓
- 是否已经过热或过度拥挤
- 是否存在更好的入场位置
- 是否需要提前失败退出
- 是否该做利润保护或时间退出

### 5.3 非目标（本次不做）

本设计不包含以下事项，避免 scope（范围）失控：

- 打开 live execution（实盘自动下单）
- 重写整套 `regime` 与 universe（标的池）逻辑
- 直接扩到大量新市场或新交易所
- 先做超复杂机器学习层
- 先做高频级别（分钟级）执行系统

## 6. 关键策略改动

### 6.1 改动一：`rotation` 从硬抑制改为分层放行

#### 当前问题

已知回测显示：`rotation` 并非没有 alpha（超额收益来源），但当前被 `suppression` 和统一分配逻辑压得过重，造成错杀。

#### 设计改法

不再采用简单的“开 / 关”二元逻辑，而改成三档：

1. **关闭**
2. **限额放行**
3. **正常放行**

由日频环境层决定当前档位；小时级确认层决定具体候选是否实际开仓。

#### 预期效果

- 降低对有效轮动机会的错杀
- 在中性偏强环境中恢复部分收益来源
- 避免完全放开导致回撤失控

### 6.2 改动二：给入场增加小时级二次确认

#### MVP 最小规则集与优先级

首版不追求把所有小时级想法一次做满，先固定最小规则集：

1. **过热过滤**（优先级最高）  
   若 `1h` 扩张已经显著超过允许区间，则拒绝新开仓，避免追在局部失衡段。

2. **拥挤过滤**（第二优先级）  
   若小时级衍生品 / 情绪代理显示交易已经过度拥挤，则拒绝新开仓。

3. **入场质量确认**（第三优先级）  
   在未被前两项否决时，检查当前价格位置是否仍具合理赔率；不再为“方向对”自动放行“位置差”的交易。

4. **4h 仅作上下文过滤**（不是并列主规则）  
   `4h` 首版只用于更高层过滤，不直接单独生成执行动作。

若同一候选同时命中过热与拥挤，按“过热优先记录、拥挤作为附加原因”处理，避免多个拒绝原因互相覆盖。

#### 当前问题

现有系统很多亏损并不是“大方向看错”，而是：

- 开得太晚
- 追在局部扩张后面
- 买在拥挤晚段
- `short` 追在容易被反抽的位置

#### 设计改法

日频层给出候选后，小时级确认层只做有限而高价值的过滤：

- 短线过热过滤
- 拥挤交易过滤
- 更优入场位置确认
- `short` 的挤压 / 反抽风险确认

#### 预期效果

- 减少“方向对、价格错”
- 改善同方向交易的平均入场质量
- 降低追在局部顶部 / 底部的概率

### 6.3 改动三：退出系统改为小时级主导

#### MVP 最小规则集与优先级

首版退出规则只做四类，且优先级固定：

1. **失败退出**（最高优先级）  
   开仓后短时间内未按预期运行，或关键小时级失效条件被破坏，则直接 `exit`。

2. **拥挤退潮退出**（第二优先级）  
   若衍生品 / 情绪拥挤显著退潮，且继续持有的赔率明显下降，则直接 `exit`。

3. **时间退出**（第三优先级）  
   若持仓拖延过久却未形成有效推进，则退出，避免资金长期占用。

4. **利润保护减仓**（第四优先级）  
   浮盈达到保护阈值后，允许一次性 `reduce`，把仓位主动降下来，避免利润大幅回吐。

若同一小时级 bar 同时出现多个退出信号，按上述优先级只执行最高级动作，不叠加多个动作。

#### 当前问题

当前系统的 `entry`（入场）体系明显比 `exit`（退出）成熟。很多交易即使方向没错，也因为退出慢、利润回吐或失败认错不够快而拉低风险调整后收益。

#### 设计改法

为各策略腿补小时级退出规则，优先增加四类：

1. **失败退出**：开仓后未按预期运行，尽快认错
2. **时间退出**：走得太慢，释放资金给更高质量机会
3. **利润保护**：浮盈后上抬保护位，减少利润回吐
4. **拥挤退潮退出**：衍生品与情绪条件恶化时提前降档或离场

#### 预期效果

- 改善回撤与资金曲线质量
- 提高可保留收益，而不是只看浮盈峰值
- 更符合交易系统而不是纯日频筛选器的行为

### 6.4 改动四：`short` 收编为条件性工具

#### 当前问题

当前 `short` 是结构上占位，但现阶段更像系统性拖累来源，而不是稳定赚钱腿。

#### 设计改法

给 `short` 增加两层门：

**第一层：日频环境门**
只有在以下环境才允许 `short` 进入候选：

- 明确 `risk-off`
- 明确 `crash`
- 明确 `cascade`
- 或明确弱结构延续环境

**第二层：小时级执行门**
即使环境允许，也必须额外过滤：

- 容易被反抽打脸的位置
- 拥挤做空位置
- 已经跌太深、赔率变差的位置

#### 预期效果

- 保留 `short` 作为防守工具
- 避免其在正常环境中持续拖累收益
- 降低被 squeeze（轧空）反杀的概率

### 6.5 改动五：分配器改成分腿预算

#### 当前问题

当前不同策略腿共享过多统一压制逻辑，导致：

- `trend` 这条主赚钱腿可能吃不满资源
- `rotation` 被过度压制
- `short` 在不成熟阶段仍可能占用不该占的风险预算

#### 设计改法

把分配器改成三条独立预算通道：

- `trend_budget`
- `rotation_budget`
- `short_budget`

再由环境层对三条预算上限做动态控制。

#### 预期效果

- 让高质量腿优先获得资源
- 让辅助腿有生存空间但不越权
- 让组合风险控制更贴近真实策略质量

## 7. 模块级落地设计

### 7.1 新增 / 扩展的数据合同

需要明确形成以下数据合同，避免不同层之间各说各话：

#### 日频候选 → 小时级确认

每个候选至少输出：

- `symbol`
- `strategy_leg`（trend / rotation / short）
- `daily_setup_type`
- `daily_score`
- `daily_invalidation`
- `regime_context`
- `leg_budget_context`
- `candidate_created_at`

#### 小时级确认 → 分配器

每个通过确认的候选至少新增：

- `entry_timing_state`
- `intraday_overheat_state`
- `crowding_state`
- `micro_invalidation`
- `execution_quality_score`
- `approval_reason` / `rejection_reason`

#### 小时级退出 → 组合状态

每个持仓至少新增：

- `exit_mode`
- `protection_state`
- `time_in_trade`
- `exit_trigger_reason`
- `crowding_unwind_state`

### 7.2 单元边界

#### Unit A：DailyRegimeBudgeter

**职责：** 输出日频环境标签与三条策略腿预算上限。  
**输入：** 日频 `regime`、衍生品摘要、市场广度。  
**输出：** `trend_budget` / `rotation_budget` / `short_budget`、策略腿开关档位。  
**依赖：** 现有 `regime` 输出。  
**不负责：** 具体入场时机与退出时机。

#### Unit B：DailyCandidateEmitter

**职责：** 继续基于日频结构生成三类候选。  
**输入：** 现有日频信号引擎。  
**输出：** 候选列表及最小共享字段。  
**依赖：** `trend` / `rotation` / `short` 引擎。  
**不负责：** 小时级微观过滤。

#### Unit C：IntradayConfirmationGate

**职责：** 对日频候选做小时级确认与拒绝。  
**输入：** 日频候选、1h / 4h 市场数据、小时级衍生品状态。  
**输出：** 通过确认的候选、拒绝原因。  
**依赖：** 小时级数据读取层。  
**不负责：** 组合预算分配。

#### Unit D：LegAwareAllocator

**职责：** 按策略腿独立预算做组合分配。  
**输入：** 日频预算、小时间级确认后的候选、风险约束。  
**输出：** 各腿最终分配结果。  
**依赖：** 现有 allocator / risk gate。  
**不负责：** 判断候选是否过热或拥挤。

#### Unit E：IntradayExitManager

**职责：** 小时级退出与保护。  
**输入：** 持仓、小时级价格 / 衍生品状态、原始 setup 信息。  
**输出：** `hold | reduce | exit` 三类确定性动作。  
**依赖：** 持仓状态、小时级数据。  
**不负责：** 新开仓筛选。

`reduce`（减仓）的 MVP 语义固定如下：

- 单次触发默认减当前仓位的 **50% notional（名义仓位）**
- 同一持仓在“从开仓到平仓”的生命周期内，`reduce` 最多触发 **1 次**
- 若之后再次出现更强退出信号，则直接进入 `exit`，不重复多次 `reduce`
- 执行优先级固定为：`hard risk gate / hard stop > exit > reduce > allocator rebalance`
- `IntradayExitManager` 负责给出动作建议；真正执行仍由现有执行 / 组合状态链路承接

#### Unit F：StrategyBacktestAttributor

**职责：** 做多版本对照、分腿归因与滚动验证。  
**输入：** 回测配置、日频与小时级数据、策略运行结果。  
**输出：** 绩效、回撤、换手、分腿贡献、稳定性报告。  
**依赖：** 现有回测链路。  
**不负责：** 实时交易决策。

## 8. 数据流

### 8.0 时间语义与事件顺序

多粒度系统最容易出错的地方不是规则本身，而是时间对齐。MVP（最小可用版）在时间语义上统一采用以下硬规则：

1. **日频信号只在日线收盘后生成**  
   每个交易日的 `daily` 候选只能使用该日已经收盘的日线 bar（K 线）与对应的日频衍生品摘要，不允许在日线未收盘时提前生成候选。

2. **小时级只读取已收盘 bar**  
   `1h` 与 `4h` 层都只能读取已经收盘的 bar；禁止在回测或运行时使用正在形成中的小时级 bar。

3. **日频先决，小时级后验确认**  
   若某个日频候选在时间点 `T_d_close` 生成，则它最早只能从 `T_d_close` 之后的第一个已收盘 `1h` bar 开始进入确认流程。小时级层绝不能读取 `T_d_close` 之后尚未收盘的数据，也不能反过来影响该日频候选是否存在。

4. **4h 在 MVP 中只作上下文过滤，不与 1h 对等决策**  
   首版先固定为：`1h` 是主执行层，`4h` 只提供更慢的上下文过滤（例如更高层趋势未破坏、未进入明显极端扩张），避免首轮实现同时维护两个并列执行层。

5. **回测事件循环固定**  
   每个交易日按以下顺序推进：
   - `daily close`：生成日频环境与候选
   - 后续每个已收盘 `1h` bar：运行小时级确认 / 退出
   - 若小时级确认通过：在该 `1h` bar 关闭后、下一可执行时点记录开仓
   - 若小时级退出触发：在该 `1h` bar 关闭后、下一可执行时点记录减仓或离场

6. **禁止 look-ahead（未来函数）**  
   任一层都不得访问其当前事件时点之后的数据；`daily` 看不到未来小时级数据，`1h` 看不到未收盘的 `1h/4h` bar，也看不到未来日线归纳结果。

### 8.1 开仓数据流

1. 日频环境层先判定当前环境与策略腿预算
2. 日频候选层输出 `trend` / `rotation` / `short` 候选
3. 小时级确认层逐个判定候选：
   - 是否过热
   - 是否拥挤
   - 位置是否太差
   - `short` 是否有挤压风险
4. 分腿分配层在各自预算内选择最终要开的单
5. 执行层按现有 `paper execution` 链路记录与下发模拟动作

### 8.2 持仓管理数据流

1. 持仓进入小时级退出层
2. 小时级退出层检查：
   - 失败退出
   - 时间退出
   - 利润保护
   - 拥挤退潮退出
3. 若触发动作，写入组合状态与运行摘要
4. 结果进入回测和归因层，用于评估退出规则质量

## 9. 回测与防过拟合设计

### 9.1 必须回答的问题

本设计落地后，回测必须至少能回答：

1. 小时级确认是否真的改善入场质量，而不是只减少交易数
2. 小时级退出是否真的减少回撤，而不是只过早止盈
3. `rotation` 从硬抑制改成分层放行后，是恢复收益还是放大噪音
4. `short` 收编为条件性工具后，是否停止系统性拖累
5. 分腿预算是否让 `trend` 贡献更集中、组合更合理

### 9.2 最小对照矩阵

至少比较以下版本：

1. **现行日频版（baseline）**
2. **日频 + 小时级确认**
3. **日频 + 小时级退出**
4. **日频 + 小时级确认 + 小时级退出**
5. **版本 4 + rotation 分层放行**
6. **版本 5 + short 条件化**
7. **版本 6 + 分腿预算**

### 9.3 必看指标

除总收益外，必须强制输出：

- 最大回撤
- Sharpe / Sortino 或等价风险调整后指标
- Calmar 或收益 / 回撤比
- 换手率
- 胜率
- 盈亏比
- 平均持仓时长
- 各策略腿收益贡献
- 样本内 / 样本外差异
- walk-forward（滚动验证）稳定性

### 9.4 防止参数炼丹的硬规则

- 先回答结构问题，再调参数问题
- 每次只允许修改少量关键自由度
- 任何新增规则都必须做 ablation（消融对照）
- 禁止只凭单窗口或单行情截图下结论
- 明确区分研究样本与验证样本
- 重要结论必须看 walk-forward，而不是只看总样本

## 10. 错误处理与保护边界

### 10.1 数据缺失处理

若小时级数据或小时级衍生品数据缺失，MVP 固定采用**单一路径**处理：

- 新开仓候选一律标记为 `cannot_confirm_intraday`
- 该候选本周期**不得开仓**
- 不自动退回“保守模式通过”，避免把数据缺口误当成放行理由
- 已有持仓若缺退出所需小时级数据，则保持上一保护状态不升级，并显式记录 `exit_data_missing`，供回测与运行报告追踪

### 10.2 层级冲突处理

若日频层与小时级层冲突：

- 日频层拥有“是否允许该腿参与”的最终否决权
- 小时级层拥有“此刻是否值得执行”的最终否决权
- 任一层否决，都不执行开仓

### 10.3 风险预算保护

- 任一策略腿都不得越过其预算上限
- `short` 默认上限必须低于 `trend`
- `rotation` 在限制模式下不得自动升级为正常模式，必须由环境层显式放开

### 10.4 状态可解释性

所有候选与持仓动作都必须留下：

- 为何通过 / 为何拒绝
- 被哪一层拒绝
- 属于哪条策略腿
- 预算是否触顶
- 退出触发原因

否则后续回测归因与实盘解释都会失真。

## 11. 推荐实施顺序

按价值 / 风险比排序，建议实施顺序如下：

1. `rotation`：硬抑制 → 分层放行
2. 小时级入场确认
3. 小时级退出系统
4. 分腿预算分配器
5. 条件性 `short`
6. 统一回测归因与 walk-forward 验证

这样排的原因是：

- 前三项最直接影响收益 / 回撤比
- 第四项决定资源如何向高质量腿倾斜
- 第五项负责把现有拖累收编成防守工具
- 第六项负责避免整个过程退化成“感觉更合理”

## 12. 实施完成后的验收标准

只有当以下条件满足，才算这套设计真正成立：

| 维度 | 验收门槛（相对 baseline） |
|---|---|
| 风险调整后收益 | Sharpe / Sortino / Calmar 中至少 **2 项改善 >= 10%** |
| 总收益 | 总收益不得下降超过 **5%** |
| 最大回撤 | 最大回撤需改善 **>= 10%**，或至少不得恶化超过 **3%** |
| 换手 | 换手上升不得超过 **20%**；若超过，则风险调整后收益必须改善 **>= 15%** 才算可接受 |
| 样本外稳定性 | walk-forward 样本外风险调整后指标需改善 **>= 5%**，且样本外总收益不得下降超过 **8%** |
| 分腿贡献 | `trend` 仍为主贡献腿；`short` 的负贡献需缩小到 baseline 的 **50% 以内**，或转为非负贡献 |
| 可解释性 | 候选、分配、退出各层都必须输出结构化理由字段，不允许出现“通过但无原因” |

若只满足收益提升，但未满足上表中的回撤 / 稳定性 / 可解释性门槛，则不得判定为完成。
