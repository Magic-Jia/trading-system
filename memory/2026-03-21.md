# 2026-03-21

- 用户指出：后台 Codex 任务结束后，Claw 没有主动发正式状态汇报，而是错误地把系统自动出现的 `Exec completed` 提示当成了已完成对老板的汇报。
- 已明确固化新规则：OpenClaw / 系统自动提示（`Exec completed`、event、commit-trigger 等）不算 Claw 已主动汇报；只要后台 Codex 任务退出，必须立即由 Claw 亲自发送固定格式的人话状态，不能等老板来问。
- 本次问题本质不是工具缺失，也不是权限问题，而是汇报流程执行失职；后续退出后允许的动作仍只限于：检查新 commit、整理固定状态消息、立刻发送给老板，然后才能做后续分析/重试。
- 用户再次复盘确认：之前误以为“子会话结束会自动通知主会话再向老板汇报”的链路可默认依赖，这是错误假设。以后默认不依赖任何自动 completion push / system event / runtime event 形成最终用户通知，必须由 Claw 把事件显式转换为正式汇报。
- 防再犯要求已收紧：若后台 Codex 已退出，而老板先看到系统提示、却没先看到 Claw 的正式状态消息，就视为 Claw 再次失职。
- 用户更新开发偏好：以后不要让 Codex 自己决定做什么改动；改动方向由 Claw 先判断和决定，再交给 Codex 执行；除非老板明确要求，否则不必每次先征求确认。
- 用户要求把“为什么老是退出后没主动汇报”的根治方案写进规则与记忆：后续面向老板的持续开发任务，停止默认依赖 `exec background` 做主执行/主回报链路，改为优先使用 OpenClaw `sessions_spawn` 的子会话 / ACP 会话来承载执行，再由主会话负责正式状态汇报。
